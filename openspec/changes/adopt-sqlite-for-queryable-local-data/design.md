## Context

FastProxy 当前已经建立了本地配置仓库、运行时编译、规则集索引和前端配置工作台。大多数用户可编辑资源仍以独立 JSON 文件保存，这对调试和备份友好；但节点缓存、规则仓库索引、健康检查历史和操作记录会持续增长，并且天然需要分页、搜索、去重、聚合和跨资源查询。

规则集索引已经迁入 SQLite，证明后端可以在不引入 CGO 依赖的情况下使用 `modernc.org/sqlite`。这次变更将 SQLite 从单点优化提升为明确的本地查询存储策略。

## Goals / Non-Goals

**Goals:**

- 明确 JSON 与 SQLite 的职责边界，避免所有数据都塞入单一存储形态。
- 将大量、历史型、索引型和关联型数据迁入 SQLite。
- 为节点缓存、健康检查、操作日志、自定义仓库索引和运行时快照建立可分页、可搜索、可测试的查询模型。
- 保持前端 API 面向业务资源，而不是暴露数据库细节。
- 允许分阶段迁移，每个数据域完成后移除对应旧缓存或旧查询路径。

**Non-Goals:**

- 不把小型用户可编辑配置整体迁入 SQLite。
- 不要求一次实现完整数据分析或复杂报表。
- 不要求引入外部数据库服务、CGO SQLite 驱动或 ORM。
- 不要求兼容已经被新模型替换的临时 JSON 缓存格式。

## Decisions

### 1. 使用 SQLite 作为可查询数据层，JSON 继续保存小型 source 配置

SQLite 承载具备以下特征的数据：数量可能大、需要分页、需要搜索、需要历史记录、需要统计或需要跨资源关联。JSON 继续承载用户直接编辑或审查的小型配置，如 profile 装配关系、基础运行参数和手写资源定义。

选择这种双层模型的原因：

- JSON 对用户可读、易备份、易人工修复。
- SQLite 对大列表、索引、历史和聚合查询更稳定。
- 一刀切迁移所有配置会增加调试成本，也会让简单资源变得不透明。

备选方案是把所有 repository 资源都迁到 SQLite。这会简化存储技术栈，但会牺牲当前开发阶段很重要的可读性和可替换性，因此不采用。

### 2. 数据域按能力独立建表，但共享数据库生命周期

后端使用同一个本地 SQLite 数据库文件承载多个数据域。每个数据域拥有自己的表、索引、查询函数和测试，不共享隐式 JSON blob 查询逻辑。数据库初始化由 Store 统一完成，schema 变更以幂等 DDL 或版本化迁移推进。

这样做的原因：

- 单库便于备份、清理和打开诊断。
- 独立表能让每个领域维持清晰边界。
- 共享初始化避免多个模块各自打开数据库造成锁竞争。

备选方案是每类数据一个 SQLite 文件。这能降低单库表数量，但会让事务、备份和跨域诊断更麻烦。

### 3. 节点缓存以“来源 + 归一化节点 + 去重键”为核心

订阅刷新和节点导入后，原始来源仍作为资源配置保存，解析后的节点进入 SQLite。节点表保存来源 ID、归一化协议字段、展示字段、去重键、标签、最后更新时间和禁用状态。前端节点列表通过分页和搜索 API 读取，不一次性拉取全部节点。

这样做的原因：

- 订阅节点数量可能很大。
- 节点去重、来源追踪和分组选择需要结构化字段。
- 原始订阅内容和可编辑 source 配置不应与查询缓存混在一起。

### 4. 健康检查历史分为原始样本和摘要视图

每次测速/健康检查写入样本表，包含节点 ID、检查类型、延迟、成功状态、错误摘要和时间。查询层提供最近状态、滚动失败率和延迟摘要。必要时保留原始样本 TTL 或按数量裁剪策略，避免数据库无限增长。

备选方案是只在节点对象上保存最后一次测速结果。它实现简单，但无法支撑趋势、失败率和问题回溯，因此只适合作为派生摘要，不作为唯一记录。

### 5. 操作事件日志统一记录资源操作和运行时操作

刷新、导入、编译、激活、核心启动、核心重载、失败回滚等都写入事件表。事件包含级别、类型、资源类型、资源 ID、关联 profile/core、消息、错误码和小型 JSON context。前端诊断页通过时间、级别和资源筛选。

这样做的原因：

- 用户遇到“配置为什么变了/为什么启动失败”时需要时间线。
- 事件日志比散落文件日志更容易关联资源。
- 事件 context 可以保存必要摘要，但不能保存 secrets 或完整大配置。

### 6. 仓库文件索引复用规则集索引经验

内置 MetaCubeX 规则集索引、自定义 GitHub 仓库文件树和未来可浏览远程仓库都使用 SQLite 存目录、文件和搜索文本。目录接口必须分页返回直接子目录和直接文件；搜索接口返回有限数量结果，并允许按仓库、核心、format、behavior 等条件扩展。

### 7. 运行时快照保存摘要，不保存无限完整配置

每次编译/预览可以记录快照摘要，包括 profile、core、状态、错误摘要、输入资源版本、输出文件路径、hash 和时间。完整 active/last-good 配置仍由 runtime 文件系统负责，SQLite 只保存便于查询和 diff 定位的元数据。若需要保存配置片段，必须限制大小并排除敏感字段。

## Risks / Trade-offs

- [SQLite schema 变更变复杂] → 建立幂等初始化和后续版本化迁移入口，每个能力配套存储测试。
- [数据库持续增长] → 为健康检查样本、事件日志和快照历史定义保留策略。
- [JSON 与 SQLite 出现双写不一致] → 明确 SQLite 中节点缓存、仓库索引、健康检查和事件是派生/查询数据；用户配置仍以 JSON 为 source-of-truth。
- [前端误用全量接口导致卡顿] → 大列表 API 必须分页，搜索接口必须限制默认返回量和最大返回量。
- [SQLite 锁竞争] → 后端 Store 统一持有数据库连接，设置单写连接和 busy timeout，长时间网络刷新不在事务中执行。

## Migration Plan

1. 保留已完成的规则集 SQLite 索引实现，并将 schema 管理抽象成可扩展的本地查询库初始化流程。
2. 迁移自定义仓库文件索引，与内置规则集索引共享目录、文件、搜索查询模型。
3. 迁移订阅节点缓存和导入节点索引，前端节点列表改用分页查询。
4. 增加健康检查历史和最近状态摘要，运行时或测速模块写入样本。
5. 增加操作事件日志，先覆盖刷新、导入、编译、激活和核心错误。
6. 增加运行时快照摘要表，用于预览/激活历史和 last-good 诊断。
7. 移除每个已迁移领域对应的旧 JSON 缓存、内存全量加载和兼容分支。

回滚策略：

- SQLite 数据属于本地缓存/历史数据，失败时不能破坏 JSON source 配置。
- 单个能力迁移失败时，可以回滚该能力的查询路径并保留 JSON source 数据。
- 数据库文件损坏时，系统应能重新初始化空库，并允许用户通过刷新订阅、刷新仓库和重新编译恢复派生数据。

## Open Questions

- 健康检查历史默认保留多久，按天裁剪还是按每节点样本数裁剪。
- 运行时快照是否需要保存脱敏后的配置片段，还是只保存 hash、路径和摘要。
- 节点缓存是否要支持全文搜索 FTS5，还是先使用普通索引与 LIKE 查询。
- 事件日志是否需要前端独立页面，还是先嵌入 Runtime/Config Management 的诊断区域。
