## Context

FastProxy 目前已经具备本地配置仓库、运行时编译和 `config-management` 入口，但规则集部分仍停留在占位状态，而且现有后端 `rule-set` 资源存的是路由规则列表，与前端表格试图表达的 sing-box `rule_set` 资源完全不是一个概念。与此同时，用户已经明确需要两类能力并存：

- 规则仓库管理：内置仓库 + 用户自定义仓库
- 规则集资产管理：用户可直接新增单条规则集，也可从仓库中挑选文件生成规则集

这次变更横跨数据模型、API、运行时编译和前端配置工作台，是一个典型的跨模块建模调整。约束包括：

- 同一条逻辑规则集必须同时支持 `sing-box` 与 `mihomo` 的不同实现。
- 管理面板只负责维护“可用规则集资产池”，不直接决定最终运行配置。
- 路由规则只能引用逻辑规则集，不能直接耦合到底层仓库分支、路径或 URL。
- 运行时需要根据激活 profile 的当前内核自动选用匹配 variant，并在缺失 variant 时给出明确错误。

## Goals / Non-Goals

**Goals:**

- 建立“规则仓库”和“规则集资产”两层模型，消除现有 `rule-set` 语义冲突。
- 支持内置仓库和用户自定义仓库，并能按 `sing-box` / `mihomo` 维护分支映射和文件浏览信息。
- 支持用户直接创建规则集资产，或从仓库选择文件快速生成规则集资产。
- 让路由规则引用逻辑规则集资产，并由运行时编译层按内核解析底层 variant。
- 为后续继续扩展更多仓库和规则文件类型保留模型空间，而不再写死到单一核心格式。

**Non-Goals:**

- 本次变更不要求完成所有 `mihomo` 运行时渲染细节，只需要把规则集解析链路和模型打通。
- 本次变更不要求实现仓库全文搜索、批量导入整仓规则或复杂的本地文件监视。
- 本次变更不要求提供通用规则文件内容编辑器，规则集内容仍以“引用已有来源”为主。
- 本次变更不要求兼容现有错误语义的 `rule-set` API 结构，允许在开发阶段直接替换为新模型。

## Decisions

### 1. 将“仓库来源”和“规则集资产”建模为独立资源

系统新增两类一等资源：

- `RuleSourceRepository`: 描述规则仓库来源
- `ManagedRuleSet`: 描述用户可在路由规则中引用的逻辑规则集

这样做而不是把“仓库字段”直接塞进规则集对象的原因：

- 仓库是可复用来源，规则集是可引用资产，生命周期和复用方式不同。
- 用户可能直接创建单条规则集，不经过仓库。
- 后续支持多个仓库下选择同名规则集时，来源与资产拆开更容易缓存、去重和扩展。

备选方案是只有一个 `RuleSet` 模型，内部可选带仓库字段；这种方案短期字段更少，但很快会演变出大量“来源模式”分支和 UI 条件判断，因此放弃。

### 2. 逻辑规则集使用“单对象 + 多内核 variant”模型

`ManagedRuleSet` 只表达逻辑身份，例如 `openai`、`cn`、`private`，实际按内核可拥有多个 `variants`。每个 variant 至少记录：

- `core`
- `sourceMode` (`repository-file` / `remote` / `local` / `inline`)
- `repositoryId`、`ref`、`path` 或 `url`
- `format`、`tag`、`behavior`、`updateInterval` 等核心相关字段

这样用户在路由规则里只需要选一次逻辑规则集，而不需要关心当前核心。运行时编译器再根据当前内核选择对应 variant。

备选方案是为每个核心分别管理一套独立规则集列表；这会导致路由规则和 UI 都必须感知当前核心，无法满足“管理一次，到处引用”的体验目标。

### 3. 路由规则只保存逻辑规则集引用，不保存底层 tag/path

路由规则层改为保存 `managedRuleSetIds` 或等价逻辑引用，而不是继续直接保存裸 `rule_set` tag。编译时再把逻辑规则集解析为目标核心需要的 tag/provider 引用。

这样做的原因：

- 同一逻辑规则集在两个核心下底层 tag、路径和格式可能不同。
- 直接保存底层 tag 会让路由规则无法跨核心复用。
- 逻辑引用更适合在前端做选择器、启用校验和资产缺失提示。

### 4. 仓库浏览采用“按内核映射分支，再读取树”的接口

`RuleSourceRepository` 为每个核心保存分支映射，例如内置 `MetaCubeX/meta-rules-dat`：

- `sing-box -> sing`
- `mihomo -> meta`

前端选择仓库后，不直接输入裸分支，而是选择目标核心，后端再按映射返回该核心对应分支下的目录树。这样可以减少用户心智负担，也能让内置仓库和自定义仓库复用同一套浏览 API。

备选方案是仅让用户手填 URL/path 模板；这种方式更简单，但无法提供“从仓库挑文件生成规则集”的核心体验。

### 5. 运行时编译新增规则集解析阶段，并按核心分流渲染

当前 runtime compiler 会解析 profile、node set、group set 和旧的 rule set。新方案中需要增加一层：

1. 读取路由规则中引用的逻辑规则集
2. 对每个逻辑规则集查找与当前核心匹配的 variant
3. 为 `sing-box` 渲染 `route.rule_set` 和规则中的 `rule_set`
4. 为 `mihomo` 渲染对应 `rule-providers` 与规则引用
5. 若缺失当前核心 variant，则编译失败并返回可读诊断

这一步必须在运行时完成，而不是在管理页预展开，因为最终使用哪一种核心只有 active profile 才知道。

### 6. 直接替换现有错误语义的 `rule-set` 资源，而不保留双轨兼容

仓库规则已经明确处于开发阶段，且仓库指南要求避免长期保留平行旧新逻辑。因此这次变更直接把现有后端 `rule-set` 资源和前端占位表迁移到新语义，删除“规则列表式 rule-set”和“资源式 rule_set”并存的中间状态。

这样做的收益：

- 避免后续实现继续围绕错误语义打补丁
- 降低前后端字段命名和状态映射的复杂度
- 为路由规则页和运行时编译统一术语

## Risks / Trade-offs

- [现有 `rule-set` 资源已有调用方，直接替换会带来破坏性迁移] → 在同一变更中同步更新前端类型、API 封装、运行时编译与路由规则页，避免半迁移状态。
- [用户只配置了一种核心 variant，切换到另一核心时编译失败] → 管理页清晰展示支持的核心集合，编译错误中明确指出缺失的规则集和目标核心。
- [仓库目录结构并不统一，自动推断格式或行为可能出错] → 第一版允许自动推断，但所有推断结果都可在规则集编辑弹窗中手动覆盖。
- [mihomo 的规则集字段与 sing-box 差异较大] → 先把 variant 模型和编译接口抽象好，把无法稳定推断的核心专属字段保持为显式配置项。
- [路由规则改为逻辑引用后，现有页面需要一次性调整较多状态结构] → 先在状态层建立“逻辑规则集选择器”与“编译期展开”分界，减少组件间重复映射逻辑。

## Migration Plan

1. 新增规则仓库与逻辑规则集的持久化目录和 API，同时保留变更开发分支内对旧 `rule-set` 资源的代码读取能力仅用于迁移脚本。
2. 将前端规则集管理页切换到新 API，并移除现有静态占位表与错误语义的本地类型。
3. 将路由规则页保存结构改为逻辑规则集引用，更新选择器与展示层。
4. 在运行时编译中接入 variant 解析，先打通 `sing-box` 输出，再接入 `mihomo` 渲染分流。
5. 删除旧 `rule-set` 资源实现及其残留引用，确保仓库内只保留新模型。

回滚策略：

- 在开发过程中可整体回滚该 change，而不是尝试保留旧新双轨。
- 运行时层继续沿用现有 active/last-good 机制，避免规则集解析失败时破坏最后一次可用配置。

## Open Questions

- `mihomo` variant 是否要在第一版就暴露完整 `behavior/path/type` 配置，还是先支持仓库文件与远程 URL 两种常见路径。
- 仓库树接口是否只做 GitHub，还是同步预留 `http-index` / `git` 抽象但延后实现。
- 逻辑规则集的 `tag` 是否默认要求跨核心一致，还是允许两个核心各自独立覆盖并由编译层做映射。
