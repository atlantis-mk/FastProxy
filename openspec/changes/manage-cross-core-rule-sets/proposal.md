## Why

FastProxy 当前把“规则集”同时当成路由规则列表和 sing-box `rule_set` 资源来使用，语义已经冲突，无法支撑“仓库管理规则来源、规则集管理可引用资产、运行时按内核选择对应实现”这条完整链路。现在需要把规则来源、逻辑规则集资产和运行时渲染彻底拆开，才能同时支持 `sing-box`、`mihomo`、内置仓库和用户自定义规则集。

## What Changes

- 新增规则仓库管理能力，用于维护内置仓库和用户自定义仓库，并为不同内核声明各自的分支或来源映射。
- 新增逻辑规则集资产管理能力，支持用户直接新增规则集，或从已配置仓库中选择文件生成规则集。
- 为单条规则集引入按内核区分的 variant 模型，让同一条逻辑规则集同时承载 `sing-box` 与 `mihomo` 的不同实现。
- 新增路由规则对逻辑规则集资产的引用能力，避免页面直接暴露底层文件路径、分支名或远程 URL。
- 新增运行时规则集解析能力，在编译配置时根据当前激活内核选择匹配的 variant，并生成对应核心格式的规则集配置。
- **BREAKING**: 现有仓库中的 `rule-set` 资源语义将从“直接持有规则列表的资源”演进为“可引用规则集资产”，相关 API、前端状态和运行时编译流程需要同步调整。

## Capabilities

### New Capabilities
- `rule-source-repositories`: 管理内置与用户自定义规则仓库，并按内核声明分支、根目录和浏览能力。
- `managed-rule-sets`: 管理用户可引用的逻辑规则集资产，支持手动创建、从仓库选取和为不同内核维护 variant。
- `kernel-aware-rule-set-resolution`: 在路由规则和运行时编译阶段按当前内核解析逻辑规则集，并渲染为目标核心可用的规则集配置。

### Modified Capabilities
- None.

## Impact

- 后端会新增规则仓库、逻辑规则集和规则集 variant 的持久化模型、CRUD API、仓库树浏览接口以及运行时解析逻辑。
- 前端 `config-management` 的规则集页会从静态占位表升级为“仓库管理 + 规则集管理”的真实工作台，路由规则页也会改为选择逻辑规则集资产而不是裸 tag。
- 运行时编译层会增加按内核选择规则集 variant 的步骤，并分别输出 `sing-box` 与 `mihomo` 需要的规则集配置片段。
- 现有 `rule-set` 相关类型、API 命名与页面状态需要统一迁移，避免“规则列表”和“规则集资源”两套语义继续并存。
