## Context

FastProxy 当前由一个 Vue 前端和一个 Go 后端骨架组成。前端仍以 zashboard 的“远程 Clash Controller 面板”心智为主，后端则已经具备本地数据目录、核心清单、运行状态和 `profile` JSON 存储等基础能力，但这些能力仍围绕“单 profile 大配置”展开，尚未建立可组合的源配置模型，也没有真实的核心 supervisor。

这次变更是一个跨前后端、跨数据模型和运行时管理方式的架构升级。目标不是把现有单个配置编辑器做大，而是把 FastProxy 提升为本地聚合内核 VPN 管理器：一边维护多个来源的配置碎片，一边在运行前编译出唯一生效的运行时配置，并由 supervisor 负责核心生命周期和热生效。

约束包括：

- V1 要优先支持 `sing-box`，但不能把数据模型写死成 sing-box 专属结构。
- 所有用户可编辑配置都需要直接落盘为 JSON 文件，便于调试、备份和后续迁移。
- 运行时只允许存在一个当前生效配置快照，避免多份活动配置造成状态漂移。
- 配置变更后需要尽可能实时生效，但 FastProxy 应用本身不能因为核心重载而重启。

## Goals / Non-Goals

**Goals:**

- 建立独立的本地配置仓库，分别管理订阅、节点集、规则集、分组集和 profile。
- 定义统一内部模型，把 Clash 订阅、纯节点链接和手动节点归一化到同一套语义上。
- 建立配置编译层，从激活的源配置集合生成唯一运行时配置。
- 建立核心 supervisor，负责启动、停止、监听运行时配置变化并仅重载核心进程。
- 在前端新增“配置工作台”心智，同时保留现有运行面板的核心价值。

**Non-Goals:**

- V1 不要求完整支持 `mihomo` 的全部配置语义和 UI 管理能力。
- V1 不要求实现云同步、多设备协同或远程 FastProxy 集群管理。
- V1 不要求做通用文本级配置编辑器，仍以结构化管理为主。
- V1 不要求增量编译或局部热更新，允许每次变更都全量重建运行时配置。

## Decisions

### 1. 采用“源配置仓库 + 唯一运行时快照”的双层模型

系统将所有用户管理内容视为 source-of-truth，并按配置类型分别存储在独立 JSON 文件中。核心不直接读取这些 source 文件，而只读取编译输出的唯一运行时快照。

选择这个模型而不是继续扩展单 `profile.content` 的原因：

- 单大文件不利于订阅更新、手动覆盖、规则复用和 profile 复配。
- source 与 runtime 分离后，失败回滚和变更审计更容易做。
- 未来支持多核心时，可以在保持同一 source 仓库的情况下切换 runtime 渲染器。

备选方案是仍然把每份 profile 视作完整配置并在界面上做局部编辑；这条路实现快，但会让订阅导入、节点去重和规则拆分在模型层持续打架，因此放弃。

### 2. 统一内部模型先于内核格式

Clash 订阅、纯节点和手动节点都先映射到统一内部模型，再由编译器面向目标核心渲染运行时配置。统一模型至少覆盖节点、节点组、规则集、运行参数、来源元数据和启用关系。

选择这一层的原因：

- 导入来源异构，但管理界面和后续合并逻辑需要一致语义。
- sing-box 优先不代表以后只能服务 sing-box。
- 去重、覆盖、优先级和来源追踪都应发生在统一模型层，而不是下放给核心配置格式。

备选方案是直接把 Clash 订阅转成 sing-box 运行时 JSON 再做小修小补。这样虽然短期可用，但 profile 组合、局部覆盖和未来 mihomo 支持都会被绑死。

### 3. profile 只存“装配关系”，不存大段运行内容

新的 profile 将描述：

- 选中的核心
- 启用的订阅、节点集、规则集、分组集
- 运行参数，如端口、日志级别、DNS/TUN 开关等
- 合并策略和优先级

profile 不直接持有全部节点、规则或最终配置文本。这样 profile 更像场景方案，可引用共享资源，并支持多个 profile 复用同一批 source 文件。

### 4. 运行时编译采用“全量编译 + 原子替换 + last-good 回滚”

每次影响运行时的变更都触发一次全量编译，而不是在现有 runtime 上打补丁。编译成功后先写入临时文件，再通过原子替换更新 `runtime/active`；supervisor 只监听该激活文件。若核心启动或重载失败，则回退到最近一次成功快照。

这样做的原因：

- 全量编译最容易保证最终结果可推导，减少状态残留。
- 原子替换可以避免 watcher 读到半写入文件。
- last-good 回滚是 V1 里对稳定性收益最大的保护。

备选方案是直接覆盖 active 文件并立即通知核心；这种方式实现简单，但失败时用户会同时失去旧配置和新配置的可运行状态。

### 5. supervisor 负责核心生命周期，FastProxy Server 负责业务编排

`internal/core` 将从当前状态占位器演进为 supervisor，负责：

- 核心二进制路径校验
- 启动、停止、重载核心进程
- 监听运行时 active 文件变化
- 在需要时做 debounce、串行重启和失败回滚

业务服务层负责：

- 存储 source JSON
- 管理 profile 激活关系
- 调用编译器生成 runtime
- 记录状态并对前端暴露 API

这样可以避免“配置存储逻辑”和“进程控制逻辑”相互污染。

### 6. 前端采用“本地工作台优先，运行面板复用”的演进式重构

现有前端有大量围绕 Clash Controller API 的页面和状态管理。V1 不适合一次性重写全部运行面板，因此采用双区策略：

- 保留现有 `Overview / Proxies / Rules / Connections / Logs` 的运行监控价值
- 新增 `Profiles / Subscriptions / Nodes / Groups / Rules / Runtime` 的配置工作台
- 前端默认连接本地 FastProxy Server，而不是把“填写远程 host/port/password”作为主入口

这样能够先交付本地产品闭环，再逐步替换旧 API 假设。

## Risks / Trade-offs

- [统一模型设计过薄，后续无法承载 mihomo] → 先把模型抽象在“节点/分组/规则/运行参数/来源元数据”层，不直接复刻 sing-box 字段。
- [Clash 订阅转换存在语义损失] → V1 明确支持边界，保留导入诊断与不兼容项提示。
- [频繁编辑触发连续重编译和重载] → 在编译触发与 supervisor 响应两层都增加 debounce 和串行队列。
- [运行时文件写坏导致核心不可用] → 使用临时文件、原子替换和 last-good 回滚。
- [前端仍旧依赖大量 Clash 风格 API] → 先新增 FastProxy 自有 API，不强求首批就替换全部运行面板数据来源。
- [profile、source、runtime 三层关系让学习成本变高] → 在 UI 中明确“来源”“场景”“当前生效配置”三种角色，并提供运行时预览。

## Migration Plan

1. 保留现有 `profiles/` 和 `runtime/state.json` 能力，新增 source 仓库目录与新 profile 结构。
2. 引入统一内部模型与编译器，但初期只为 `sing-box` 渲染 active runtime。
3. 将运行时管理从占位状态机切到真正 supervisor，同时保持 API 兼容性。
4. 前端先增加新的配置管理页面和 bootstrap 数据，再逐步将入口改成本地 FastProxy 模式。
5. 对旧 `profile.content` 提供一次性迁移或导入路径，避免现有测试数据直接失效。
6. 在 active runtime 首次稳定运行后，再扩展更多内核能力和高级编辑能力。

回滚策略：

- 保留旧 profile 文件直到迁移确认完成。
- supervisor 维护最近一次成功的 runtime 快照，重载失败时自动回退。
- 如需功能级回滚，可停用新工作台入口，仅保留旧运行面板和旧 profile 存储。

## Open Questions

- `mihomo` 在 V1 中是只保留抽象和路径配置，还是要支持最小可启动能力。
- Clash 订阅导入后，是否要保留原始响应快照以便重新解析和问题排查。
- 手动编辑规则与分组时，是否允许覆盖订阅生成内容，还是必须通过“派生副本”编辑。
- 唯一运行时文件的物理格式是否统一为 JSON，还是允许按核心输出不同格式但逻辑上只有一个 active runtime。
